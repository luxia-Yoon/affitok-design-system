Visual Hierarchy (시각 계층)

모든 화면은 세 개의 Zone으로 구성된다. 각 Zone은 역할이 다르며, 비중과 타이포그래피도 다르게 적용한다.

Zone 규칙 (Z1–Z5)

규칙설명
Z1Hero Zone은 화면당 하나만 존재한다. 두 개 이상이면 시선이 분산된다.
Z2Hero → Context → Action 순서를 유지한다. 역순은 허용하지 않는다.
Z3Context Zone의 텍스트 크기는 Hero보다 항상 작다 (최소 4px 차이).
Z4Action Zone CTA는 화면당 Primary 1개 + Secondary 최대 2개로 제한한다.
Z5모달·다이얼로그도 동일한 3-Zone 구조를 따른다.

Data Density (데이터 밀도)

밀도는 정보량과 사용 컨텍스트에 따라 세 단계로 나뉜다. 같은 페이지 안에서 밀도를 혼용하지 않는다.

밀도 선택 기준

조건권장 처리
행 수 10개 초과그룹핑(섹션/탭)을 추가한다
숫자 컬럼 3개 이상 인접컬럼 헤더에 단위 레이블을 명시한다
컬럼 5개 초과기본 3~4개만 노출하고 나머지는 "열 설정"으로 숨긴다
텍스트 셀이 길어질 때최대 2줄로 제한하고 말줄임(truncate)을 적용한다
액션이 행마다 필요할 때hover 시 인라인 액션을 표시한다 (항상 노출 금지)
모바일 화면Comfortable 밀도 고정, Compact 금지

Toss Product Principles (토스 제품 원칙)

토스팀이 제품 결정에 사용하는 6가지 판단 기준. 기능을 추가하거나 UI를 결정할 때 이 원칙으로 검증한다.

1

익숙한가? (Familiar)

사용자가 이미 알고 있는 패턴을 활용한다. 새로운 UI는 배움 비용을 발생시킨다.

레이어점검 질문예시
언어이 단어를 사용자가 이미 아는가?"정산" vs "인센티브 지급"
구조이 레이아웃이 다른 앱에도 있는가?좌측 사이드바 네비게이션
행동이 인터랙션이 직관적인가?스와이프로 삭제, 탭으로 선택
2

3초 안에 답할 수 있는가? (Easy to Answer)

화면을 보고 3초 안에 다음 행동을 알 수 없다면 설계를 다시 한다.

Do — 한 화면에서 사용자가 해야 할 행동이 명확하다.

캠페인에 지원하시겠어요?

지원하기
Don't — 선택지가 너무 많아 어디를 눌러야 할지 모른다.

다음 단계를 선택하세요

지원하기
찜하기
공유하기
신고하기
나중에 보기
3

이유를 설명하는가? (Why)

사용자에게 어떤 행동을 요청하거나 제한할 때 반드시 이유를 제공한다.

상황이유 없음 (나쁨)이유 있음 (좋음)
정보 수집"전화번호를 입력해주세요""본인 인증에 사용됩니다"
기능 제한"이 기능을 사용할 수 없습니다""파트너 승인 후 사용 가능합니다"
대기 안내"처리 중...""광고 데이터를 불러오는 중 (최대 10초)"
권한 요청"알림을 허용해주세요""캠페인 승인 알림을 받을 수 있어요"
4

한 메시지만 전달하는가? (One Thing)

한 화면, 한 메시지, 한 CTA. 두 개 이상이면 어느 것도 기억되지 않는다.

CTA 규칙기준
Primary CTA화면당 1개만 허용. 가장 중요한 다음 행동.
Secondary CTA최대 2개. Primary보다 시각적으로 약하게.
Tertional 이하텍스트 링크 또는 메뉴 진입으로 처리.
Destructive Action항상 별도 확인 단계(다이얼로그) 추가.
5

대기를 제거하는가? (No More Loading)

로딩 스피너는 "우리가 느리다"는 신호다. 인지적 대기를 줄이는 전략을 우선한다.

전략설명적용 상황
Skeleton UI레이아웃 형태를 먼저 보여준다리스트, 카드, 테이블
Optimistic Update서버 응답 전에 UI를 먼저 반영한다좋아요, 상태 변경, 삭제
Progressive Disclosure중요 데이터 먼저 → 부가 데이터 나중에메인 수치 → 차트 → 상세
Prefetch사용자가 이동할 가능성이 높은 페이지를 미리 로드상세 페이지 링크 hover
Inline Feedback0.1초 내 시각 반응 (버튼 눌림 등)모든 인터랙티브 요소
6

핵심만 남기는가? (Minimum Features)

추가할 기능을 고민하기 전에 제거할 기능을 먼저 찾는다.

체크포인트기준 질문
사용 빈도지난 30일간 이 기능을 쓴 사용자가 10% 이상인가?
대체 가능성다른 기능이 이것을 대체할 수 있는가?
학습 비용새 사용자가 이 기능을 혼자 이해할 수 있는가?
유지 비용이 기능을 유지하는 데 월 X시간 이상 쓰는가?
핵심 목표 기여이것이 사용자의 핵심 목표 달성에 직접 기여하는가?

Error Message Principles (에러 메시지 원칙)

에러는 사용자 여정이 끊기는 순간이다. 명확한 원인과 해결책을 동시에 제공해야 한다.

Dark Mode Rules (다크모드 원칙)

다크모드는 밝기를 낮추는 것이 아니라 빛을 시뮬레이션하는 것이다. 레이어가 높을수록 밝아진다.

6대 핵심 규칙 (DM1–DM6)

DM1순수 흰색 배경 금지

#FFFFFF 배경은 눈부심을 유발한다. 최소 bg-[#0A0A0A] 사용.

DM2순수 검은색 금지

#000000은 OLED 번인 위험. bg-[#0A0A0A] ~ bg-[#141414] 사용.

DM3레이어 = 밝기

모달 > 카드 > 페이지 순서로 표면이 밝아진다.

DM4채도 낮춤 필수

다크 배경에서 채도 100% 색상은 과도하게 진동한다. 채도를 20~30% 낮춘다.

DM5텍스트 불투명도

본문 white/80, 보조 white/60, 힌트 white/40. 순수 흰색은 제목만.

DM6그림자 색 변경

다크모드 그림자는 rgba(0,0,0,0.4) 이상. 밝은 그림자는 오히려 이상하다.
